Yes — **absolutely**.
What you described works **just as well with virtual machines** as with physical nodes, and this is actually a *very common* way to test or run on-prem Kubernetes setups.

The key point is:

> **BGP does not care whether a node is physical or virtual** — it only cares that something can advertise routes and receive packets.

Below is how this typically looks and what you need to watch out for.

---

## High-level architecture (VM-based)

```
Client
  ↓
Router / Gateway (BGP peer)
  ↓
Virtual Machines (Kubernetes nodes)
  ↓
Kubernetes Gateway / Services
  ↓
Pods
```

Your **Vagrant VMs** become the Kubernetes nodes.

---

## What you need (minimum)

### 1. VMs with real L3 connectivity

Your Vagrant VMs must:

* Be on a network where the router can reach them
* Have stable IPs
* NOT be NAT-only

**Use one of these Vagrant networking modes:**

* `public_network` (bridged)
* Routed network via your hypervisor
* A lab router VM (VyOS, FRR) between host and VMs

🚫 Avoid default NAT — BGP will not work there.

---

### 2. A router that speaks BGP

This can be:

* Your **physical gateway** (UniFi, pfSense, MikroTik)
* A **router VM** (VyOS, FRR)
* A Linux box running FRRouting

This router will peer with your Kubernetes nodes.

---

### 3. Kubernetes + MetalLB (BGP mode)

Inside the cluster:

* Install **MetalLB**
* Configure BGP peering between MetalLB speakers and the router
* Allocate a pool of service IPs

Each VM will run a MetalLB speaker.

---

### 4. Gateway API controller (optional but typical)

Examples:

* Envoy Gateway
* HAProxy Gateway
* NGINX Gateway
* OpenShift Router (if you’re actually running OpenShift)

This handles L7 routing **after** traffic enters the cluster.

---

## Example: Vagrant setup (conceptual)

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "ubuntu/jammy64"

  (1..3).each do |i|
    config.vm.define "k8s-node#{i}" do |node|
      node.vm.hostname = "k8s-node#{i}"
      node.vm.network "public_network", ip: "192.168.50.1#{i}"
      node.vm.provider "virtualbox" do |vb|
        vb.memory = 4096
        vb.cpus = 2
      end
    end
  end
end
```

These VMs can now:

* Peer with your router via BGP
* Advertise service IPs
* Receive traffic directly

---

## Traffic flow in this VM setup

1. Service or Gateway gets IP `192.168.50.100`
2. MetalLB advertises `192.168.50.100/32` via BGP
3. Router installs route:

   ```
   192.168.50.100 → k8s-node2
   ```
4. Traffic is forwarded directly to that VM
5. Kubernetes routes to pods

No NAT, no port-forwarding.

---

## Common VM gotchas (important)

### 1. Hypervisor networking must allow BGP

* VirtualBox bridged mode: ✅ works
* libvirt bridge: ✅ works
* VMware bridge: ✅ works
* Vagrant NAT: ❌ will fail

---

### 2. Reverse path filtering (Linux)

Disable or loosen it on the VMs:

```bash
sysctl -w net.ipv4.conf.all.rp_filter=0
```

---

### 3. MTU consistency

Nested virtualization sometimes breaks PMTU.

* Use consistent MTU
* Avoid jumbo frames in labs

---

### 4. IP pool must be routable

MetalLB IPs must:

* Be in a subnet your router knows
* Not overlap DHCP ranges
* Not be “floating” without routing

---

## Does this scale / is it “real”?

Yes. This is effectively how:

* On-prem Kubernetes clusters are tested
* CI environments emulate bare metal
* Vendors validate BGP + MetalLB
* People build “cloud-like” labs

The only thing you *don’t* get is:

* Hardware offload
* DDoS absorption
* Internet-scale Anycast

Everything else is real.
